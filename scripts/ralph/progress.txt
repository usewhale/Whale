## Codebase Patterns
- Cross-platform shell behavior should live in `internal/shell` with injectable `GOOS` and `LookPath` dependencies so tests do not depend on the host platform.
- Windows shell resolution should prefer `pwsh`, then `powershell.exe`, and should not fall back to `cmd.exe`, Git Bash, MSYS Bash, `bash`, or `sh`.
- Foreground `exec_shell` should resolve the shell before creating `exec.Cmd`; resolver failures are returned as `shell_unavailable` tool errors without starting a process.
- Background `exec_shell` tasks should resolve the shell before creating `exec.Cmd`; resolver failures are recorded on the task as `failed` with stderr set to the resolver error and nil exit code.
- Hooks should resolve the shell before creating `exec.Cmd`; resolver failures should return `SpawnErr` so blocking hook events preserve existing block semantics.
- Runtime-facing prompt text should use injectable `goos` and shell display name helpers so Windows/Linux assertions do not depend on the host platform.
- Runtime-facing tool descriptions should use injectable `goos` helpers and shared shell display names so Windows-only guidance does not leak into Unix descriptions.
- Shell read-only approval prefixes should be selected through a GOOS-testable helper so Windows PowerShell commands can be allowed without leaking into Unix behavior.
- User-facing policy and CLI wording should use neutral "shell command" terminology; Bash-specific names should stay out of approval prefix surfaces.
- Shell tool hook documentation should match `exec_shell`; Whale does not translate hook commands between Unix shell syntax and PowerShell syntax.
- Release workflow builds run from an Ubuntu matrix; Windows assets should be cross-compiled with `GOOS=windows`, `GOARCH=amd64`, `CGO_ENABLED=0`, uploaded as both `.exe` and `.zip`, with the zip staging filename renamed to `whale.exe`.
