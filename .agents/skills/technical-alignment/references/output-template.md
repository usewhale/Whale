# Output Template

Use this template when converting the `claude-proposal-consensus` `APPROVED` final consensus plan into `03-technical-alignment.md`.

```markdown
# [功能名] 技术对齐稿

## 目标
- [来自完整业务需求文档的一句话技术落地目标，不重新扩展需求]

## 事实来源
- 代码路径：[关键页面、组件、接口、服务、模型、迁移、测试文件；无则写“无”]
- 文档来源：[AGENTS、architecture / codebase 文档、explore 结果；无则写“无”]
- 核验结论：[已确认的当前实现事实，不写未核验猜测]

## 当前实现事实
- 前端现状：[入口、页面、组件、状态、关键交互]
- 后端现状：[接口、服务、权限、任务或事件]
- 数据现状：[表、字段、状态、枚举、历史数据]
- 测试现状：[已有相关单元、接口、前端或回归测试入口；无则写“无”]
- 差异：[当前行为与目标行为的关键差异]

## 技术方案
- 前端：[承载入口、组件、关键交互、状态和失败反馈]
- 后端：[入口、服务边界、调用链、权限校验点]
- 接口：[请求、响应、错误行为、兼容要求]
- 数据：[数据库结构、字段、状态 / 枚举、迁移 / 历史数据策略]
- 权限 / 安全边界：[前端展示边界与后端兜底边界]

## 数据库变化
- 结论：[有数据库结构或迁移变更 / 无数据库结构或迁移变更]
- 表 / 字段：[新增、修改、删除的表字段；无则写“无”]
- 状态 / 枚举：[状态字段、枚举值、流转变化；无则写“无”]
- 迁移 / 兼容：[历史数据、默认值、nullable、回滚或兼容策略；无则写“无”]

## 复用与不采用
- 复用：[明确复用的现有能力]
- 不采用：[明确不走的方案及原因]

## 数据流 / 状态流
- 数据流：[关键业务流程的数据流]
- 状态流：[状态字段或业务状态如何变化；无则写“无”]
- 失败流：[失败时数据和 UI / 调用方如何保持一致]

## 影响范围
- 会影响：[页面、接口、服务、角色、状态]
- 不影响：[明确不应改动的模块或行为]

## 风险与约束
- [兼容性、历史数据、迁移、权限、性能、并发等真实风险]
- [AI 开发必须遵守的实现约束]

## 验证方向
- [单元、接口、前端、回归或手工验收方向；无自动化入口也要说明]

## 需要对齐的问题
1. [只放会改变实现方向、复用边界、数据 / 权限 / 接口方案或验收可落地性的问题]
   推荐：[推荐答案与理由]
```

## 压缩规则

- 技术对齐稿必须定位为已确认技术对齐文档，包含事实来源、当前实现事实、技术方案、数据库变化、数据流 / 状态流、复用与不采用、影响范围、风险与约束、验证方向。
- 目标必须来自完整业务需求文档，不重新发明需求。
- 事实来源必须写清代码路径、文档来源和核验结论，用于支撑后续技术合同；缺少当前实现事实或事实来源时必须阻塞。
- 当前实现事实应来自 `explore` 结果及必要本地核验；必须写清前端现状、后端现状、数据现状、测试现状和当前行为与目标行为的关键差异，不输出大段代码清单。
- 技术方案必须来自 `claude-proposal-consensus` 的 `APPROVED` 最终共识方案；必须写清前端承载入口、后端入口、接口请求 / 响应 / 错误 / 兼容、数据结构、状态 / 枚举、权限、数据流、状态流、失败流和验证方向，足以支撑后续技术合同展开。
- 数据库变化必须固定单列；涉及数据库变化时，不允许只混写在后端、服务、技术方案或数据模型描述里；无数据库结构或迁移变更时也必须明确写“无”。
- 影响范围必须同时写“会影响”和“不影响”，明确实现边界。
- 权限 / 安全边界必须区分前端展示边界与后端兜底边界。
- “不采用”要写清楚原因，供阅读者理解方案取舍。
- 只有会改变实现方向、复用边界、数据 / 权限 / 接口方案或验收可落地性的问题才放到 `需要对齐的问题`，最多 3 个；没有就写“无”。
- 不复述完整业务需求文档，不写背景铺垫，不写“我将会”。

## 保存前质量门禁

- 必须有事实来源、当前实现事实、技术方案、数据库变化、接口、权限 / 安全边界、数据流 / 状态流、影响范围、风险与约束、验证方向。
- 事实来源必须能追溯到代码路径、项目文档、`explore` 结果或必要本地核验结论。
- 数据库变化必须单列；无变化也必须写“无数据库结构或迁移变更”。
- 权限 / 安全边界必须区分前端展示边界与后端兜底边界。
- 影响范围必须同时写“会影响”和“不影响”。
- `需要对齐的问题` 最多 3 个；没有则写“无”。
- 不允许把待确认问题写成确定方案。
- 缺少事实来源、当前实现事实或 Claude `APPROVED` 技术方案时必须阻塞。
