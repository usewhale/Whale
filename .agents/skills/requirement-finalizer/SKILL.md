# Requirement Finalizer

把“原始需求 / 原始文档”和已确认的业务对齐稿合并成一份新的完整业务需求文档。此 skill 只写业务内容，不写技术方案，不拆执行任务，不生成实现批次，不修改业务代码。

核心目标：**产出一份完整、自洽、可直接阅读的业务需求文档，读者不需要再回看原始需求或业务对齐稿。**

## 输入

- 必需：原始需求、原始文档内容，或用户给出的原始材料路径
- 必需：已确认的业务对齐稿内容，默认路径为 `docs/需求文档/<feature-name>/01-scenario-alignment.md`
- 可选：用户明确给出的功能名、保存路径、需要保留的业务命名或补充说明

如果缺少原始材料或业务对齐稿，先要求用户补齐；不要只基于单边材料生成完整业务需求文档。

## 路径规则

- 业务对齐稿默认读取：`docs/需求文档/<feature-name>/01-scenario-alignment.md`
- 完整业务需求文档默认输出：`docs/需求文档/<feature-name>/02-business-requirement.md`
- 如果用户显式指定输出路径，以用户指定路径为准
- 如果需要从路径提取功能名，优先使用 `docs/需求文档/<feature-name>/` 的目录名

## 工作流程

1. 读取输入材料：
   - 读取原始需求 / 原始文档内容
   - 读取业务对齐稿
   - 识别功能名、目标角色、业务入口、对象、状态、流程、规则和验收口径
2. 合并业务内容：
   - 保留原始需求中已经明确的业务内容
   - 吸收业务对齐稿中确认后的业务口径、范围、场景、验收和边界
   - 原始材料和业务对齐稿冲突时，以对齐后确认的业务口径为准，并直接写最终结论
3. 补齐读者理解所需的业务说明：
   - 对容易误解的业务名词、角色、状态、按钮、入口、边界条件直接解释清楚
   - 把省略、跳转引用、口语化描述改写成完整正文
   - 用初学者也能理解的清楚语言，减少阅读者反复追问
4. 检查输出是否自洽：
   - 读者不需要打开原始需求或业务对齐稿也能理解要做什么
   - 验收口径是用户能观察到的业务结果
   - 文档中没有来源文件路径、引用地址或“见某文档”式替代说明
5. 保存完整业务需求文档；如果仍有阻塞业务结论的问题，最多提出 1-3 个问题并附推荐答案，不强行收口。
6. 成功保存完整业务需求文档后，必须触发 `claude-consensus` skill 进行文件复审；传入原始需求 / 原始文档和 `01-scenario-alignment.md` 业务对齐稿作为只读参考，`02-business-requirement.md` 作为唯一可写目标，并按下方“Claude Consensus 复审”审核源需求、业务对齐稿、最终需求的三方一致性。
7. 只有复审结论为 `APPROVED` 后，才能以复审后的完整业务需求文档作为下一阶段输入；如果复审结论为 `BLOCKED`，必须停止流程，向用户说明阻塞点，并等待用户补充或确认。

## 输出格式

```markdown
# [功能名] 完整业务需求文档

## 业务目标
- [用业务语言说明本功能要达成的结果]

## 适用角色
- [角色名]：[这个角色在本功能中的含义、权限或关注点]

## 业务入口与对象
- 入口：[用户从哪里进入或触发]
- 对象：[本功能处理的业务对象，以及对象含义]

## 业务范围
- 包含：[本次明确要做的内容]
- 不包含：[本次明确不处理的内容；没有则写“无”]

## 业务流程
1. [用户可理解的步骤和系统应表现出的业务行为]
2. ...

## 业务规则
- [状态、权限、可见性、生成 / 审核 / 保存 / 下载、默认值、历史数据、异常边界等规则]

## 验收口径
- [用户能观察到的业务结果，不写技术实现细节]

## 需要确认的问题
1. [只放会阻塞最终业务结论的问题]
   推荐：[推荐答案与理由]
```

## 输出要求

- 文档必须是一份新的完整业务需求文档，不是原文摘要，也不是对齐稿补丁。
- 保留原始需求中已经明确的业务内容；不要因为业务对齐稿更精炼而丢失原始材料里的明确要求。
- 对齐稿已经确认的业务口径、范围、场景、验收和边界必须吸收进最终文档。
- 原始材料与对齐稿冲突时，以对齐后确认的业务口径为准，不在正文里展开“双方冲突”过程。
- 对容易误解的业务名词、角色、状态、按钮、入口、边界条件，要在文档里直接解释。
- 验收口径只写用户能观察到的结果，不写技术实现、数据库、接口或组件细节。
- 没有待确认问题时，`需要确认的问题` 写“无”。

## Claude Consensus 复审

本 skill 只要成功产出并保存完整业务需求文档，必须触发 `claude-consensus` skill 进行文件复审。不要把底层脚本作为主入口；主入口是调用 `claude-consensus` skill。复审通过前，完整业务需求文档不得作为下一阶段输入。

复审必须同时提供三类文档上下文：

- 原始需求 / 原始文档：只读参考，不允许修改。
- `01-scenario-alignment.md` 业务对齐稿：只读参考，不允许修改。
- `02-business-requirement.md` 完整业务需求文档：唯一允许修改的文件。

给 `claude-consensus` 的任务口径应明确：

```text
复审 requirement-finalizer 的产物。

输入：
- 源需求：<source-requirement-path-or-content>，只读参考
- 业务对齐稿：<scenario-alignment-path>，只读参考
- 完整业务需求文档：<business-requirement-path>，唯一可修改文件

要求：
- 只允许修改完整业务需求文档。
- 不允许修改源需求和业务对齐稿。
- 检查完整业务需求文档是否完整覆盖源需求中的有效信息。
- 检查完整业务需求文档是否忠实继承业务对齐稿中的已确认业务口径。
- 检查是否引入源需求和业务对齐稿都未确认的新范围。
- 检查是否遗漏约束、非目标、异常、角色、状态、业务规则和验收口径。
- 检查是否把待确认事项写成确定需求。
- 通过后，完整业务需求文档才能作为下一阶段业务基准。
```

复审结论处理：

- `APPROVED`：以复审后的完整业务需求文档作为下一阶段输入。
- `APPROVED_WITH_NOTES` 或 `REVISE`：由 `claude-consensus` subagent 根据 Claude 复审反馈只修改完整业务需求文档，并继续复审直到 `APPROVED` 或 `BLOCKED`。
- `BLOCKED`：不要继续进入下一阶段；向用户说明阻塞点，并等待用户补充或确认。

## 禁止内容

- 不要在完整业务需求文档中写入原始文档地址、对齐稿地址、来源文件路径。
- 不要写“按原始需求”“根据对齐稿”“见某某文档”“同上文档”等引用替代正文说明。
- 不要输出技术方案、接口设计、数据库设计、组件拆分、实施顺序、user stories 或执行 story。
- 不要直接基于零散原始需求跳过业务对齐稿完成收口。
- 不要把未确认的冲突内容同时保留为两个可选方案；应先提出问题或写入已确认结论。
- 复审时不要允许 `claude-consensus` 修改源需求或业务对齐稿。
