# Technical Contract

基于完整业务需求文档和已确认技术对齐文档，生成可直接指导初级开发实现的最终完整技术方案合同。合同必须消除方案歧义，固化实现入口、接口契约、数据变化、状态流、权限边界、兼容策略、异常处理和验证标准。

此 skill 只做最终技术合同收口：继承 `02-business-requirement.md` 和已确认技术对齐文档，不重新探索方案，不扩展业务需求，不写正式 PRD，不写 user stories，不拆执行批次，不修改业务代码，不替用户确认未确认选项。

`technical-contract` 不依赖 `technical-alignment` skill 的执行历史；换句话说，不依赖 technical-alignment skill：不检查上游 skill 是否运行过，不限定技术对齐文档的生成者，只检查输入的已确认技术对齐文档是否内容完整、已确认、可展开为合同。

## 输入

- 必需：已确认的完整业务需求文档。可以来自用户显式传入的文档地址、默认路径 `docs/需求文档/<feature-name>/02-business-requirement.md`，或用户直接粘贴的完整文档内容。
- 必需：已确认技术对齐文档。可以来自用户显式传入的任意文档地址、默认路径 `docs/需求文档/<feature-name>/03-technical-alignment.md`，或用户直接粘贴的完整技术对齐内容。
- 可选：用户明确给出的功能名、保存路径、必须保留的技术约束或命名。

如果找不到完整业务需求文档或已确认技术对齐文档，要求用户提供；不要根据零散原始需求、单独业务对齐稿、未确认方案或缺少事实来源的技术讨论直接生成技术合同。

## 路径规则

- 完整业务需求文档优先读取用户显式传入的文档地址。
- 如果用户没有显式传入地址，优先读取：`docs/需求文档/<feature-name>/02-business-requirement.md`。
- 如果用户直接在对话中提供完整业务需求文档内容，可以直接使用该内容。
- 已确认技术对齐文档优先读取用户显式传入的文档地址。
- 如果用户没有显式传入地址，优先读取：`docs/需求文档/<feature-name>/03-technical-alignment.md`。
- 如果用户直接在对话中提供完整技术对齐文档内容，可以直接使用该内容。
- 技术合同默认输出：`docs/需求文档/<feature-name>/04-technical-contract.md`。
- 如果需要从路径提取功能名，优先使用 `docs/需求文档/<feature-name>/` 的目录名。

## 匹配规则

关联文档时使用确定性匹配，不猜测：

- 优先从用户明确给出的功能名、`docs/需求文档/<feature-name>/` 目录名或技术合同文件名 `docs/需求文档/<feature-name>/04-technical-contract.md` 提取 `<feature-name>`。
- 完整业务需求文档匹配路径：`docs/需求文档/<feature-name>/02-business-requirement.md`。
- 技术对齐稿匹配路径：`docs/需求文档/<feature-name>/03-technical-alignment.md`。
- 技术合同匹配路径：`docs/需求文档/<feature-name>/04-technical-contract.md`。
- 如果只找到唯一候选且上下文明确指向同一功能，可以使用该候选。
- 如果存在多个候选、功能名无法确定、或完整业务需求文档、已确认技术对齐文档与技术合同无法对应到同一 `<feature-name>`，必须停止并询问用户，不得猜测。

## 输入完整性检查

生成合同前，必须先检查已确认技术对齐文档是否已经明确以下内容：

- 当前实现事实。
- 事实来源。
- 前端入口、页面 / 组件承载位置、关键交互行为。
- 后端接口、服务入口、权限校验点。
- 接口入参、出参、错误行为、兼容要求。
- 数据库结构、字段、状态 / 枚举、迁移 / 历史数据策略。
- 关键业务流程的数据流、状态流和失败流。
- 复用能力、不采用方案、影响范围、不影响范围。
- 风险、约束、验证方式。

缺少当前实现事实或事实来源时必须阻塞。缺少任何会影响开发落地的最终口径时也必须阻塞，不生成 `04-technical-contract.md`。阻塞说明必须列出缺失项，并要求补齐已确认技术对齐文档或让用户确认最终口径。不要把缺失信息猜成确定方案。

阻塞时使用以下格式：

```markdown
无法生成技术合同：已确认技术对齐文档缺少会影响开发落地的当前事实、事实来源或最终口径。

- 缺失项：[具体缺少的入口 / 接口 / 数据 / 状态 / 权限 / 兼容 / 测试口径]
- 影响：[为什么会导致开发无法直接实现]
- 需要补齐：[请补齐已确认技术对齐文档或确认最终口径]
```

## 工作流程

1. 读取并确认完整业务需求文档：
   - 只继承已确认的业务目标、范围、流程、规则和验收口径。
   - 不重新扩展需求，不改写产品口径。
2. 读取已确认技术对齐文档：
   - 接受默认 `03-technical-alignment.md`、用户显式传入的其他路径或用户粘贴的完整内容；不依赖 `technical-alignment` skill 的执行历史。
   - 提取当前实现事实、事实来源、最终技术方案、前后端入口、接口契约、数据结构、数据流、状态流、失败流、权限边界、复用与不采用、影响范围、风险、约束和验证方式。
   - 去掉未确认选项、讨论过程、备选方案和临时问题。
   - 如果技术对齐文档仍存在未确认问题，先要求用户确认；不要替用户选择。
3. 执行输入完整性检查：
   - 已确认技术对齐文档足以支撑完整合同模板时，继续生成。
   - 缺少会影响实现的最终口径时，按“输入完整性检查”阻塞，不生成合同。
4. 合并为最终完整技术方案合同：
   - 用业务需求约束目标、验收边界和非目标。
   - 用已确认技术对齐文档约束实现入口、接口、数据、状态、权限、异常、兼容、验证和不影响边界。
   - 只保留最终结论，不新增方案探索。
5. 保存技术合同；如果缺少会影响合同成立的最终结论，则阻塞，不强行生成。
6. 成功保存技术合同后，必须触发 `claude-consensus` skill 进行文件复审；传入 `02-business-requirement.md` 完整业务需求文档和已确认技术对齐文档作为只读参考，`04-technical-contract.md` 作为唯一可写目标，并按下方“Claude Consensus 复审”审核最终合同继承一致性和可开发性。
7. 只有复审结论为 `APPROVED` 后，才能以复审后的技术合同作为后续执行、拆解或代码实现的最终技术基准；如果复审结论为 `BLOCKED`，必须停止流程，向用户说明阻塞点，并等待用户补充或确认。

## 输出格式

```markdown
# [功能名] 技术合同

## 目标与验收边界
- 目标：[来自完整业务需求文档]
- 验收边界：[必须满足的业务结果]
- 非目标：[明确不做的范围；无则写“无”]

## 事实来源
- 代码路径：[来自已确认技术对齐文档的关键页面、组件、接口、服务、模型、迁移、测试文件；无则写“无”]
- 文档来源：[来自已确认技术对齐文档的 AGENTS、architecture / codebase 文档、explore 结果；无则写“无”]
- 核验结论：[已确认的当前实现事实，不写未核验猜测]

## 当前实现事实
- 前端现状：[入口、页面、组件、状态]
- 后端现状：[接口、服务、权限、任务或事件]
- 数据现状：[表、字段、状态、枚举、历史数据]
- 差异：[当前行为与目标行为的关键差异]

## 最终技术方案
- 前端：[最终承载方式和交互路径]
- 后端：[最终接口、服务、调用链]
- 数据：[最终数据结构和状态表达]
- 权限 / 安全：[前端展示边界 + 后端兜底边界]

## 前端实现合同
- 入口：[路由 / 页面 / 组件]
- 状态：[加载、空态、正常、错误、禁用、提交中]
- 交互：[用户操作、触发条件、刷新策略、失败反馈]
- 接口调用：[调用时机、参数来源、结果处理]
- 兼容：[不应破坏的现有页面行为；无则写“无”]

## 后端实现合同
- 入口：[Controller / Router / Job / Listener / Service]
- 服务边界：[新增或复用的服务职责]
- 调用链：[从入口到数据读写的顺序]
- 事务 / 幂等 / 并发：[需要则写明；无则写“无特殊要求”]
- 错误处理：[业务错误、权限错误、数据不存在、重复提交等]

## 接口合同
- 接口：[方法 + 路径 / RPC 名称]
- 权限：[访问角色、租户 / 用户边界、后端兜底校验]
- 请求：[字段、类型、必填、默认值、来源]
- 响应：[字段、类型、语义]
- 错误：[错误码 / 状态码 / 文案策略]
- 兼容：[是否影响旧调用方；无则写“无”]

## 数据库变化
- 结论：[有数据库结构或迁移变更 / 无数据库结构或迁移变更]
- 表 / 字段：[新增、修改、删除；无则写“无”]
- 状态 / 枚举：[新增或调整；无则写“无”]
- 索引 / 约束：[新增或调整；无则写“无”]
- 迁移 / 兼容：[历史数据、默认值、nullable、回滚或兼容策略；无则写“无”]

## 数据流 / 状态流
- 数据流：[用户操作 -> 前端状态 -> 接口 -> 服务 -> 数据库 -> 响应 -> UI 更新]
- 状态流：[状态字段或业务状态如何变化；无状态变化则写“无”]
- 失败流：[失败时数据和 UI 如何保持一致]

## 复用与不采用
- 复用：[明确复用的现有能力]
- 不采用：[明确不走的方案及原因]

## 影响范围
- 会影响：[页面、接口、服务、角色、状态、数据]
- 不影响：[明确不应改动的模块或行为]

## 风险与约束
- 兼容性：[旧数据、旧接口、旧页面]
- 权限：[必须后端兜底的点]
- 性能：[查询、批量、分页、缓存等；无则写“无特殊风险”]
- 并发：[重复提交、竞态、幂等等；无则写“无特殊风险”]
- 实现约束：[AI / 开发者必须遵守的边界]

## 验证合同
- 单元测试：[必须覆盖的服务、函数、规则]
- 接口测试：[成功、失败、权限、边界]
- 前端测试：[关键交互、状态、错误反馈]
- 回归测试：[不能破坏的旧流程]
- 手工验收：[按业务需求可验证的最小场景]
```

## 输出要求

- 技术合同必须包含目标与验收边界、事实来源、当前实现事实、最终技术方案、前端实现合同、后端实现合同、接口合同、数据库变化、数据流 / 状态流、复用与不采用、影响范围、风险与约束、验证合同。
- 目标、验收边界和非目标必须来自完整业务需求文档，不重新发明需求。
- 当前实现事实、事实来源、最终技术方案、复用与不采用、影响范围、风险与约束必须来自已确认技术对齐文档，不重新探索方案。
- 不在合同阶段重新探索代码，不在合同阶段补写未确认方案，不根据经验猜测当前实现事实。
- 禁止只写“新增接口”“调整服务”“更新状态”“适配”“处理”“完善”“支持”等笼统表达；出现这些词时，必须同时给出具体入口、字段、状态、错误、权限、调用链或验证口径。
- 接口必须写清方法和路径 / RPC 名称、权限、请求字段、响应字段、错误行为和兼容要求。
- 前端必须写清入口、状态、交互、失败反馈和接口调用时机。
- 后端必须写清入口、服务边界、调用链、事务 / 幂等 / 并发要求和错误处理。
- 数据库变化必须单独成章，并继承已确认技术对齐文档；涉及数据库变化时，不允许只混写在后端、服务、技术方案或数据模型描述里；无数据库结构或迁移变更时必须明确写“无数据库结构或迁移变更”，表 / 字段、状态 / 枚举、索引 / 约束、迁移 / 兼容分别写“无”。
- 数据流 / 状态流必须能让开发者理解从用户操作到数据落库再到 UI 更新的完整路径；无状态变化时也必须明确写“无”。
- 影响范围必须同时写“会影响”和“不影响”，明确实现边界。
- 权限 / 安全边界必须区分前端展示边界与后端兜底边界。
- “不采用”要写清楚不走的方案及原因，供阅读者理解实现边界。
- 验证合同必须明确到测试类型和场景，不允许只写“补充测试”。
- 技术合同正文只写最终结论，不保留未确认选项、讨论过程或“需要确认的问题”章节。
- 如果已确认技术对齐文档没有足够信息支撑这些要求，不能猜测，必须阻塞。

## Claude Consensus 复审

本 skill 只要成功产出并保存技术合同，必须触发 `claude-consensus` skill 进行文件复审。不要把底层脚本作为主入口；主入口是调用 `claude-consensus` skill。复审通过前，技术合同不得作为后续执行、拆解或代码实现的最终技术基准。

复审上下文应包含：

- `02-business-requirement.md` 完整业务需求文档：只读参考，不允许修改。
- 已确认技术对齐文档：只读参考，不允许修改；可以是默认 `03-technical-alignment.md`、用户显式传入的其他路径或用户粘贴内容。
- `04-technical-contract.md` 技术合同：唯一允许修改的文件。

给 `claude-consensus` 的任务口径应明确：

```text
复审 technical-contract 的产物。

输入：
- 完整业务需求文档：<business-requirement-path>，只读参考
- 已确认技术对齐文档：<technical-alignment-path-or-content>，只读参考
- 技术合同：<technical-contract-path>，唯一可修改文件

要求：
- 只允许修改技术合同。
- 不允许修改完整业务需求文档或已确认技术对齐文档。
- 检查技术合同是否忠实继承完整业务需求文档的业务目标、范围、规则和验收边界。
- 检查技术合同是否只消费已确认技术对齐文档内容，没有引入技术对齐文档没有确认的新事实或新方案。
- 检查技术合同是否忠实继承已确认技术对齐文档的当前实现事实、事实来源、最终技术方案、前端入口、后端入口、接口契约、数据变化、状态流、失败流、权限边界、复用与不采用、影响范围、风险、约束和验证方式。
- 检查合同是否在缺少当前实现事实、事实来源或最终口径时仍生成了确定合同；如有，必须阻塞。
- 检查技术合同是否足够让初级开发直接实现，不需要再次询问方案、接口、数据、状态、权限、兼容、异常处理或测试口径。
- 检查技术合同是否只保留最终结论，没有重新探索方案、补写备选方案或替用户确认未确认选项。
- 检查是否遗漏接口请求 / 响应 / 错误 / 权限 / 兼容要求。
- 检查是否遗漏前端入口、状态、交互、失败反馈、接口调用时机。
- 检查是否遗漏后端入口、调用链、服务边界、事务、幂等、并发和错误处理。
- 检查是否包含固定的数据库变化章节，且忠实继承已确认技术对齐文档中的数据库结构变更、字段、状态 / 枚举、索引 / 约束、迁移和历史数据兼容结论；无变化时是否明确写“无数据库结构或迁移变更”。
- 检查是否遗漏数据流、状态流、失败流或历史数据兼容。
- 检查是否遗漏权限 / 安全边界、兼容性、性能、并发或不影响范围等关键约束。
- 检查验证合同是否明确到单元测试、接口测试、前端测试、回归测试和手工验收场景。
- 检查是否存在“适配”“处理”“完善”“支持”“调整”“优化”等模糊词，但没有具体实现口径。
- 检查是否把未确认问题写成确定方案。
- 检查是否重新发明了已确认技术对齐文档中没有确认的方案。
- 通过后，技术合同才能作为后续执行和拆解的最终技术基准。
```

复审结论处理：

- `APPROVED`：以复审后的技术合同作为后续 story 拆解、代码实现或执行 agent 的最终技术基准。
- `APPROVED_WITH_NOTES` 或 `REVISE`：由 `claude-consensus` subagent 根据 Claude 复审反馈只修改技术合同，并继续复审直到 `APPROVED` 或 `BLOCKED`。
- `BLOCKED`：不要继续进入执行或拆解；向用户说明阻塞点，并等待用户补充或确认。

## 禁止内容

- 不要生成 PRD。
- 不要写 user stories、完整验收矩阵或执行 story 拆分。
- 不要生成或修改执行批次文件。
- 不要修改业务代码。
- 不要扩展业务需求。
- 不要探索新技术方案、补写备选方案或替用户确认未确认选项。
- 不要在合同阶段重新探索代码。
- 不要根据经验猜测当前实现事实。
- 不要保留待确认问题；缺少最终口径时必须阻塞。
- 不要把原始需求、零散口述需求或单独业务对齐稿当成已确认完整业务需求文档。
- 不要把未确认的技术讨论、待选方案、缺少事实来源的方案或问题清单当成已确认技术对齐文档。
- 复审时不要允许 `claude-consensus` 修改完整业务需求文档或已确认技术对齐文档。
