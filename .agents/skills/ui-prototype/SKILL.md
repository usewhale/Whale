# UI Prototype

在进入技术方案前，把需求触达的页面/界面先锁定为原型图与可交互 HTML。非技术用户只需要看图确认，不需要写任何技术内容。

此 skill 编排 `draw-ui`（生图 + HTML 还原）和 `agent-browser`（截参考图），不自行生产原型内容。

## 输入

- 必需：已确认的完整业务需求文档。可以来自用户显式传入的文档地址、默认路径 `docs/需求文档/<feature-name>/02-business-requirement.md`，或用户直接粘贴的完整文档内容
- 可选：用户上传的现有 App 截图、设计稿，或指定要参考的页面 URL
- 可选：用户指定的功能名、自定义输出路径

如果找不到完整业务需求文档，要求用户提供；不要根据原始需求、零散口述或未确认的材料直接生成原型。

## 路径规则

- 完整业务需求文档优先读取用户显式传入的文档地址
- 如果用户没有显式传入地址，优先读取 `docs/需求文档/<feature-name>/02-business-requirement.md`
- 如果用户直接在对话中提供完整业务需求文档内容，可以直接使用该内容
- 原型默认输出到 `docs/需求文档/<feature-name>/ui-prototype/`
- 如果用户显式指定输出路径，以用户指定路径为准
- 如果需要从路径提取功能名，优先使用 `docs/需求文档/<feature-name>/` 的目录名

## 输出

```
ui-prototype/
├── <screen-name>.png          （原型图，快速确认用）
├── <screen-name>.html         （HTML 还原，实现 AI 可直接参照）
└── ui-spec.md                 （UI 规格：每页区块、交互、状态描述）
```

## 工作流程

### Step 1：从需求文档分析页面结构

读取完整业务需求文档，自动分析：

- 本功能涉及几个页面 / 界面？
- 每个页面包含哪些业务内容块（列表、表单、详情区、筛选栏、图表等）
- 哪些是对现有页面的修改、哪些是全新页面

将分析结果简要告知用户，确认理解无误后继续。不需要用户逐页描述 UI 细节。

### Step 2：获取风格参考

自动获取现有 App 的视觉风格：

- **可截图的应用页面**：用 `agent-browser` 截取相关现有页面，作为风格参考图。优先截取与新功能相近的页面（如同类列表页、同类表单页）
- **用户上传了截图/设计稿**：直接作为参考图
- **项目中有设计稿文件**：扫描 `docs/` 或用户指定的路径，取相关文件
- **以上皆无**：明确告知用户"没有现有参考图，AI 产出的 UI 风格可能与实际 App 不一致，建议提供 1-2 张现有页面截图"

不做"用提示词描述风格"——AI 直接看参考图提取视觉特征。

### Step 3：生成原型图

调用 `draw-ui` 为每个页面生成原型图：

- 将 Step 2 的参考图作为 `draw-ui` 的 `--frame` / `--ref` 输入，约束风格一致
- 为每个页面构造 prompt（以 Step 1 的分析 + 需求文档中的业务内容为准）
- 使用清单法："该页面包含：xxx 筛选区、yyy 表格、zzz 操作按钮……"
- 不写像素、列数、padding 之类的排版规格
- 多页串行生成，不并行

### Step 4：用户确认原型图

展示原型图给用户。用户只需要回答两种话：

- "可以" → 进入下一步（HTML 还原）
- "改 xxx" → 调整 prompt，重新生成该页 → 再看 → 循环

不要一次展示过多信息。只展示原型图和简单的页面标注。

### Step 5：HTML 还原

原型图确认后，调用 `draw-ui` 的 HTML 还原能力：

- 按 `draw-ui` 的 `references/html-reconstruction.md` 流程执行
- 页面布局、卡片、按钮、文字、常规图标 → HTML/CSS/SVG
- Logo、品牌插画、3D/玻璃质感等难复刻元素 → 素材化
- 排版做逐项校准（字体、字重、字距、换行位置）
- 输出可独立打开的 `.html` 文件

### Step 6：用户确认 HTML

用户用浏览器打开 HTML，确认可交互效果：

- "可以" → 进入下一步（保存定稿）
- "改 xxx" → 修改 HTML/CSS → 再看 → 循环

### Step 7：保存定稿

- 原型图（png）保存到输出目录
- HTML 文件保存到同一目录
- 生成 `ui-spec.md`，包含：
  - 每个页面的名称、业务功能
  - 页面包含的区块（列表/表单/详情/图表等）
  - 交互状态（loading、empty、error、hover 等）
  - 对现有页面的改动点（如有）

保存完成后，告知用户可以继续走 `technical-alignment`。

## ui-spec.md 格式

```markdown
# [功能名] UI 原型

> 原型图与 HTML 位于同目录下。本规格供技术方案和实现阶段参考。

## 现有 App 风格参考
- 参考页面：[列表 / 详情 / 表单 / ...]
- 参考来源：[`agent-browser` 截图 / 用户上传 / 设计稿 / 无]

## Screen 1: [页面名]

- 原型图：`<screen-name>.png`
- HTML：`<screen-name>.html`
- 业务功能：[一句话]
- 页面区块：
  - [区块 1 名称]：[展示什么，可能有的交互]
  - [区块 2 名称]：[展示什么，可能有的交互]
- 交互状态：
  - loading：[有/无，描述]
  - empty：[有/无，描述]
  - error：[有/无，描述]
- 对现有页面的改动：[如有，说明改了哪里]

## Screen N: ...
```

## 用户交互设计

| 步骤 | 用户做什么 | 门槛 |
|------|-----------|------|
| Step 1 | 听 AI 说"识别到 N 个页面"，确认 | 点头 |
| Step 2 | 可选：上传截图 / 告诉 AI 去哪个页面截参考 | 零门槛 |
| Step 3 | 不需要参与 | — |
| Step 4 | 看图 → "可以" 或 "改 xxx" | 零门槛 |
| Step 5 | 不需要参与 | — |
| Step 6 | 浏览器打开 HTML → "可以" 或 "改 xxx" | 零门槛 |

全程不需要用户写任何技术文档、描述布局规格、指定像素。用户只在 AI 产出后做确认。

## 禁止内容

- 不写技术方案、接口设计、组件拆分、数据库设计
- 不修改业务代码
- 不回退到"用文字描述 UI 让用户去想象"
- 不跳过参考图获取（除非确实没有）
- 不并行生成多页（draw-ui 要求串行）
- 不把原型图 or HTML 直接当作最终实现代码
